API credentials

Register an application here to integrate against TOSTI's REST API or MCP server. You'll get a client_id — and, for confidential clients, a client_secret — that you embed in your code to drive the OAuth 2.0 flow. The client secret is only shown once at creation; copy it somewhere safe.

Flow: for any client that ships to users (mobile/desktop/SPA) use authorization code with PKCE. For server-to-server integrations with no user involved, use client credentials (requires a confidential client). When an access token expires (after 10h), exchange the refresh token at the token endpoint for a new pair — refresh tokens are rotated, so keep only the latest.

Endpoints: do not hard-code paths. Discover them from /.well-known/oauth-authorization-server (RFC 8414). For reference, today's paths are /oauth/authorize/ and /oauth/token/. Redirect URIs must be https (or nu.thalia for the Thalia app).

Scopes: request the smallest scope that does the job; users see what you ask for on the consent screen. The full list (currently read, write, orders:order, orders:manage, thaliedje:request, thaliedje:manage, transactions:write) and what each one means is in the OAuth integration guide — which also covers dynamic client registration, token introspection, revocation, and everything else not on this page.

Send the access token as Authorization: Bearer <token> on every request to /api/v1/ or /mcp.
