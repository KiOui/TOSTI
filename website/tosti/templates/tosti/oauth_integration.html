OAuth 2.0 integration guide

How to authenticate against TOSTI from your own app or AI assistant

TOSTI exposes its REST API at /api/v1/ and its MCP server at /mcp. Both speak OAuth 2.0 for authorisation. This page documents which flows TOSTI supports, how to register a client, and what the conventions are. It applies whether you're building a developer integration, a third-party app, or hooking up an AI assistant.

Endpoints

TOSTI advertises its OAuth configuration via the standard discovery documents. Fetch one of these and the rest of the URLs are inside it — you should not hard-code endpoint paths:

  • Authorisation server metadata (RFC 8414): /.well-known/oauth-authorization-server — tells you the authorize, token, revoke, and registration endpoints.
  • Protected-resource metadata (RFC 9728): /.well-known/oauth-protected-resource — tells you which authorisation server protects this resource.

For reference, the actual paths today: /oauth/authorize/, /oauth/token/, /oauth/revoke_token/, /oauth/introspect/, and /oauth/register/ (dynamic client registration). They can move — the discovery documents are the source of truth.

Which flow to use

For any integration with TOSTI you should use the authorization-code grant with PKCE. It's the flow our documentation, tooling and examples are written for, and the only one we'd recommend planning around. Refresh tokens are part of the same flow so long-running clients don't have to keep prompting the user.

Authorization code with PKCE

The standard interactive flow: redirect the user to /oauth/authorize/ with a code challenge, get a code back at your redirect URI, exchange the code (plus the code verifier) at /oauth/token/ for an access token. PKCE (RFC 7636) is required for every client. Use the S256 code-challenge method.

Refresh token

When an access token expires, exchange the refresh token at /oauth/token/ for a new pair. Refresh tokens are rotated — the response includes a new refresh token, and the old one is invalidated. Keep only the most recent one. Access tokens last 10 hours; refresh tokens do not expire on a timer but are revoked when the user revokes the application's access.

If authorization code + PKCE genuinely can't cover your use case, get in touch with the website committee before building something else.

Registering a client

Two ways to get a client_id, depending on who you are. Both produce a public client — no client secret, PKCE mandatory.

Through your account page (developers)

Log in, open My account → API credentials and click Create new application. You'll get a client_id — the public-client setup means there's no secret to copy.

Dynamic client registration (MCP clients, AI assistants)

Clients can self-register without human intervention by POSTing to /oauth/register/ per RFC 7591. Registration is rate-limited per IP (10 per hour) to protect the Application table. The issued client is provisioned with the intersection of the scopes you request and the scopes TOSTI publishes.

Redirect URIs

Redirect URIs must use one of the schemes allowed by TOSTI. Currently those are https and the nu.thalia custom scheme (for the Thalia app's deep links). HTTP is rejected; this is enforced both at static client creation and at dynamic registration. For local development you can extend the list in tosti/settings/development.py, never in base.py.

You may register multiple redirect URIs separated by whitespace, and the client must use one of them exactly when starting an authorisation request.

Scopes

TOSTI uses fine-grained scopes so users can grant only the permissions an app actually needs. Always request the smallest scope that does the job; users see your request on the consent screen and will deny a request that seems overreaching.

Tokens

Send the access token as Authorization: Bearer <token> on every authenticated request to /api/v1/ or /mcp. Access tokens are opaque (treat them as strings, do not parse them) and last 10 hours. When you get a 401 Unauthorized, refresh the token if you have a refresh token, or restart the flow.

Unauthenticated requests to /mcp and /api/ get a 401 with a WWW-Authenticate header pointing at the protected-resource metadata (RFC 9728), so a compliant client can auto-discover the authorisation server with no prior configuration.

Revoking access

You can revoke a token programmatically at /oauth/revoke_token/ (RFC 7009). Users can revoke an app's access entirely from My account → Connected apps — that wipes all the user's tokens for that application in one click.
